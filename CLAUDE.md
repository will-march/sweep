# CLAUDE.md

Guide for Claude Code (and other agents) working in this repository. Read this once at the start of a session, then trust the code.

## What this is

iMaculate is a **macOS-only Flutter desktop app** that cleans system and developer caches. It is a single-binary GUI tool — no server, no networking, no telemetry. Everything happens on local disk via Dart APIs and `osascript`.

Entry point: `lib/main.dart` → `IMaculateApp` (`lib/app.dart`) → either `SplashScreen` or `HomeShell`.

## Running locally

```bash
flutter pub get
flutter run -d macos
```

For a clean rebuild after dependency or platform changes:

```bash
flutter clean && rm -rf macos/Pods macos/Flutter/ephemeral build .dart_tool
flutter pub get && flutter run -d macos
```

Release build:

```bash
flutter build macos --release
```

## Conventions

- **Cupertino + Material mix.** UI uses Cupertino icons and aesthetic but `MaterialApp` for theming. Don't reach for `CupertinoApp` — the Aurora theme tokens assume Material's `ColorScheme`.
- **No global state library.** State is local `StatefulWidget` + `Future`/`Stream`. Keep it that way unless the user asks otherwise.
- **Async iteration over directories** must use `dir.list(followLinks: false)` — symlinks have already bitten us once.
- **Privileged work** runs through `osascript ... with administrator privileges`. Always escape with `quoted form of` and pre-escape `\` and `"` in Dart before splicing into the script (see `CacheRemover._privilegedEmpty` and `TrashService.permanentlyDelete` for the canonical pattern). Do not concatenate raw paths into shell strings.
- **Trash > permanent delete.** New deletion flows should default to `TrashService.moveToTrash`. `permanentlyDelete` exists for system caches that Finder won't trash.
- **`RiskLevel` is load-bearing.** It drives chip colour, confirmation copy, and which deletion path is taken (`CacheRemover.empty` branches on it). Every entry in `cleaning_targets.dart` must declare one.

## What NOT to delete from the repo

- `.metadata` — Flutter's project marker, regenerated on `flutter create` but easier left alone.
- `analysis_options.yaml` — lint rules, opinions intentional.
- `pubspec.lock` — committed on purpose; this is an app, not a library.
- `macos/` — the entire Xcode shell is committed (minus the ignored ephemeral bits).

## What is gitignored on purpose

- `build/`, `.dart_tool/`, `.flutter-plugins-dependencies` — regenerated by Flutter.
- `macos/Flutter/ephemeral/`, `macos/Pods/`, `macos/.symlinks/` — regenerated by `pod install` / Flutter tooling.
- `.claude/settings.local.json` — per-user Claude Code permissions; never commit.

If you need to commit something currently ignored, check `.gitignore` and propose the change explicitly rather than `-f`-adding it.

## Adding a new cache target

1. Pick the right mode in `lib/data/cleaning_targets.dart`.
2. Use a path that `path_resolver.expandPath` can handle (`~`, `$USER`, or absolute).
3. Pick the **lowest** `RiskLevel` that's honest. If the user can lose work, it's `higher`.
4. Run `flutter analyze` and at least manually scan in Light Scrub mode.

## Tests

- `flutter test` — unit tests under `test/`. Don't write tests that hit real `~/Library` paths; inject a fake `Directory` or fake `Process.run` instead.
- The `osascript` services are deliberately thin so they can be replaced by fakes.

## Things to avoid

- Don't add a network dependency. The "no network" claim in the README is a hard product constraint.
- Don't introduce `path_provider` or other plugins for things `dart:io` already does.
- Don't widen permissions past what the current admin escalation grants — re-prompt instead.
- Don't add analytics/crash reporting libraries.
- Don't write multi-paragraph comments. Code here is self-explanatory once you have the model above.

## When you're unsure

- The four cleaning modes are defined entirely in `lib/data/cleaning_targets.dart` and `lib/models/cleaning_level.dart`. Read those before adding anything to the cleaner UI.
- Tree-map performance work belongs in `lib/services/disk_scanner.dart`. UI smoothness comes from streaming `ScanProgress` events, not from caching results.
- For new screens, follow `cleaner_screen.dart` as the template — it's the cleanest example of the Aurora aesthetic + service wiring.
